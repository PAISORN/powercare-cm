# คู่มือ Workflow การพัฒนาแบบไม่กระทบเว็บจริง

เอกสารนี้ใช้สำหรับกำหนดวิธีทำงานกับโปรเจกต์ `powercare-cm` ให้ปลอดภัยต่อเว็บที่ใช้งานจริง

จากสถานะปัจจุบันของโปรเจกต์นี้:

- Git branch หลักคือ `master`
- โปรเจกต์เชื่อมกับ GitHub แล้ว
- โปรเจกต์เชื่อมกับ Vercel แล้ว
- ถ้ามีการ push ไปยัง branch production ที่ Vercel ใช้อยู่ เว็บจริงอาจ deploy อัตโนมัติได้

ดังนั้นหลังจากนี้ **ห้ามพัฒนางานต่อบน `master` โดยตรง**

## 1. เป้าหมายของ workflow นี้

เราต้องการให้:

- แก้ไขและพัฒนาระบบต่อได้อย่างปลอดภัย
- ทดสอบงานใหม่ก่อนขึ้นเว็บจริง
- แยกงานที่กำลังพัฒนาออกจาก production
- ลดความเสี่ยงที่เว็บจริงจะเปลี่ยนโดยไม่ได้ตั้งใจ

## 2. กติกาหลัก

### กติกา 1

`master` ให้ถือเป็น **Production Branch**

ความหมายคือ:

- ใช้สำหรับโค้ดที่พร้อมขึ้นเว็บจริงแล้วเท่านั้น
- ไม่แก้งานทดลองบน branch นี้
- ไม่ push งานครึ่ง ๆ กลาง ๆ มาที่ branch นี้

### กติกา 2

ทุกงานใหม่ให้ทำบน **branch แยก**

ตัวอย่างชื่อ branch:

- `feature/dashboard-improvement`
- `feature/request-form-update`
- `feature/cm-import-v2`
- `fix/tracking-date-bug`
- `fix/workflow-permission`

### กติกา 3

จะขึ้นเว็บจริงก็ต่อเมื่อ:

- งานพัฒนาเสร็จแล้ว
- ทดสอบแล้ว
- คุณอนุมัติแล้ว
- merge หรือ push เข้า `master`

## 3. วิธีทำงานที่แนะนำ

### ขั้นที่ 1: เริ่มงานใหม่จาก `master`

ทุกครั้งก่อนเริ่มงานใหม่:

1. ดึงโค้ดล่าสุดของ `master`
2. สร้าง branch ใหม่จาก `master`

แนวคิดคือ:

- `master` = ฐานที่เสถียรที่สุด
- branch ใหม่ = พื้นที่ทำงานของฟีเจอร์นั้น

### ขั้นที่ 2: พัฒนาใน branch งาน

ในช่วงพัฒนา:

- แก้ไฟล์ในเครื่องได้ตามปกติ
- commit ได้ตามปกติ
- run local test ได้ตามปกติ
- เปิด dev server ได้ตามปกติ

**ทั้งหมดนี้ยังไม่กระทบเว็บจริง**

### ขั้นที่ 3: push ขึ้น GitHub เฉพาะ branch งาน

เมื่ออยาก backup งานหรือส่งให้ Vercel สร้าง preview:

- push ขึ้น GitHub แต่ push ไปที่ branch งาน

ตัวอย่าง:

- push `feature/dashboard-improvement`
- push `fix/tracking-date-bug`

ผลที่ได้:

- โค้ดถูกเก็บบน GitHub
- ถ้า Vercel รองรับ preview deployment จะได้ลิงก์ทดสอบแยก
- เว็บจริงยังไม่เปลี่ยน

### ขั้นที่ 4: ทดสอบก่อนขึ้นจริง

ก่อน merge เข้า `master` ให้ตรวจอย่างน้อย:

- หน้าแรก
- Dashboard
- All Work
- Request
- Tracking
- Profile
- Admin
- Print document
- Responsive mobile

ถ้ามีการแตะฐานข้อมูล ให้ตรวจเพิ่ม:

- สร้างข้อมูลได้
- แก้ไขข้อมูลได้
- ปิดงานได้
- export ได้
- audit ยังทำงาน

### ขั้นที่ 5: ขึ้น production เมื่อ “สั่ง”

เมื่อคุณพร้อมจริง:

1. ตรวจ branch งานรอบสุดท้าย
2. merge เข้า `master`
3. push `master`
4. ให้ Vercel deploy production

**ขั้นนี้เท่านั้นที่ควรถือว่าเว็บจริงจะเปลี่ยน**

## 4. รูปแบบ workflow ที่แนะนำสำหรับโปรเจกต์นี้

ใช้แบบนี้ได้เลย:

### โครงสร้าง branch

- `master` = production
- `feature/...` = ฟีเจอร์ใหม่
- `fix/...` = แก้บั๊ก

### ตัวอย่างการใช้งานจริง

ถ้าจะทำหน้า dashboard ใหม่:

- สร้าง `feature/dashboard-v3`
- ทำงานทั้งหมดบน branch นี้
- push branch นี้ขึ้น GitHub
- ทดสอบ preview
- เมื่อคุณสั่งค่อย merge เข้า `master`

ถ้าจะแก้บั๊ก tracking:

- สร้าง `fix/tracking-status-layout`
- ทดสอบให้เรียบร้อย
- ค่อย merge เข้า `master`

## 5. สิ่งที่ “ไม่” กระทบเว็บจริง

รายการต่อไปนี้ **ไม่กระทบเว็บจริง** ถ้ายังไม่ไปแตะ `master`:

- แก้ไฟล์ในเครื่อง
- รัน `npm run dev`
- commit ในเครื่อง
- สร้าง branch ใหม่
- push ไป branch อื่นที่ไม่ใช่ production branch

## 6. สิ่งที่ “อาจ” กระทบเว็บจริง

รายการต่อไปนี้ต้องระวัง:

- push เข้า `master`
- merge PR เข้า `master`
- redeploy production บน Vercel
- เปลี่ยน environment variables ของ production
- เปลี่ยนฐานข้อมูล production

## 7. กฎการทำงานกับฐานข้อมูล

ถึงแม้โค้ดจะยังไม่ขึ้นเว็บจริง แต่ถ้าไปใช้ `DATABASE_URL` ของ production โดยตรง ก็อาจกระทบข้อมูลจริงได้

ดังนั้นให้แยกความเข้าใจเป็น 2 ส่วน:

### โค้ด

- ยังไม่กระทบ production ถ้ายังไม่ merge/push เข้า `master`

### ข้อมูล

- อาจกระทบทันที ถ้า script หรือหน้าเว็บที่รันอยู่ชี้ไปยัง production database

สรุป:

> โค้ดปลอดภัยได้ แต่ข้อมูลอาจไม่ปลอดภัย ถ้าใช้งานกับ production database

## 8. วิธีที่ปลอดภัยที่สุดสำหรับงานต่อจากนี้

ถ้าจะพัฒนาระบบต่อ แนะนำลำดับนี้:

1. สร้าง branch ใหม่ทุกครั้ง
2. ทำงานบน branch นั้น
3. ทดสอบ local ก่อน
4. push branch นั้นขึ้น GitHub
5. ใช้ preview URL ตรวจงาน
6. เมื่อคุณอนุมัติ ค่อย merge เข้า `master`

## 9. Checklist ก่อนขึ้น Production

ก่อน merge เข้า `master` ให้เช็กว่า:

- [ ] ฟีเจอร์เสร็จครบตามที่ต้องการ
- [ ] ทดสอบ flow หลักแล้ว
- [ ] ทดสอบ mobile แล้ว
- [ ] ไม่มี error สำคัญ
- [ ] ถ้ามี schema/db change ได้สำรองข้อมูลแล้ว
- [ ] ถ้ามี import/export ได้ลองใช้งานจริงแล้ว
- [ ] คุณอนุมัติให้ขึ้น production แล้ว

## 10. Workflow สั้นที่สุดสำหรับใช้งานจริง

จำง่าย ๆ แบบนี้:

1. `master` = เว็บจริง
2. งานใหม่ทุกงาน = ทำใน branch ใหม่
3. ทดสอบให้ผ่านก่อน
4. พร้อมเมื่อไรค่อย merge เข้า `master`

## 11. ข้อแนะนำเพิ่มเติม

ถ้าจะทำงานระยะยาว แนะนำเพิ่มในอนาคต:

- branch `staging` สำหรับทดสอบก่อน production
- Vercel preview สำหรับทุก branch งาน
- checklist release ก่อน deploy จริง
- backup database ก่อนงานใหญ่

---

## สรุปสุดท้าย

หลังจากนี้ถ้าเราทำตาม workflow นี้:

- คุณสามารถพัฒนาต่อได้เรื่อย ๆ
- เว็บจริงจะยังไม่เปลี่ยน
- เว็บจริงจะเปลี่ยนก็ต่อเมื่อคุณอนุมัติและให้ merge/push เข้า `master`

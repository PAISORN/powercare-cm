# Release Checklist ก่อนขึ้น Production

เอกสารนี้ใช้เป็น checklist ก่อนนำโค้ดขึ้นใช้งานจริงบน production ของ `PowerCare.CM`

เป้าหมายคือ:

- ลดความเสี่ยงที่เว็บจริงจะพัง
- ลดโอกาสข้อมูลผิด
- ทำให้การ deploy เป็นขั้นตอนที่ตรวจสอบได้

## 1. กติกาหลัก

ให้ใช้ checklist นี้ทุกครั้งก่อน:

- merge เข้า `master`
- push ที่จะทำให้ Vercel deploy production
- เปลี่ยนฐานข้อมูล production
- เปลี่ยน environment variables ของ production

## 2. Checklist ก่อน Release

### 2.1 ตรวจ scope งาน

- [ ] งานที่ต้องการขึ้น production ชัดเจนแล้ว
- [ ] ไม่มีงานทดลองหรือโค้ดครึ่งเสร็จปะปน
- [ ] branch ที่จะปล่อยขึ้นจริงเป็น branch ที่ถูกต้อง
- [ ] รู้ชัดว่าการเปลี่ยนแปลงครั้งนี้กระทบหน้าไหนหรือ role ไหนบ้าง

### 2.2 ตรวจโค้ด

- [ ] ไม่มี error ระหว่าง build
- [ ] ไม่มีไฟล์ชั่วคราวที่ไม่ควรขึ้น repo
- [ ] ไม่มี config ทดสอบค้างอยู่
- [ ] ไม่มี URL / secret / token หลุดในโค้ด
- [ ] ถ้ามี script ชั่วคราว ใช้เสร็จแล้วและไม่ทำงานเองโดยไม่ตั้งใจ

### 2.3 ตรวจ flow หลักของระบบ

- [ ] หน้าแรกเปิดได้ปกติ
- [ ] หน้า Dashboard เปิดได้
- [ ] หน้า All Work เปิดได้
- [ ] หน้า Request เปิดได้
- [ ] หน้า Tracking เปิดได้
- [ ] หน้า Profile เปิดได้
- [ ] หน้า Reports เปิดได้
- [ ] หน้า Admin เปิดได้ตามสิทธิ์

### 2.4 ตรวจ workflow ผู้ใช้งานจริง

#### ผู้แจ้งซ่อม

- [ ] แจ้งซ่อมได้สำเร็จ
- [ ] ได้เลขที่แจ้งซ่อม
- [ ] ติดตามงานได้ด้วยเลข CM

#### ช่าง

- [ ] login ได้
- [ ] เห็นงานของหมวดตัวเอง
- [ ] รับงานได้
- [ ] เริ่มดำเนินการได้
- [ ] ส่งรอปิดงานได้
- [ ] ปล่อยงานคืนคิวได้

#### วิศวกร

- [ ] login ได้
- [ ] เห็นงานในหมวดตัวเอง
- [ ] ส่งกลับให้แก้ไขได้
- [ ] ปิดงานได้
- [ ] พิมพ์เอกสารปิดงานได้

#### Admin

- [ ] login ได้
- [ ] สร้าง user ได้
- [ ] แก้ไข user ได้
- [ ] reset password ได้
- [ ] ลบ user ได้
- [ ] ดู history ได้
- [ ] ดู audit trail ได้
- [ ] จัดการ category / zone / SLA ได้

### 2.5 ตรวจข้อมูลและฐานข้อมูล

- [ ] schema/database change ถูกต้อง
- [ ] ถ้ามี import ข้อมูล ได้ตรวจผลแล้ว
- [ ] ข้อมูลตัวอย่างหรือข้อมูลทดสอบที่ไม่ควรอยู่ production ถูกลบแล้ว
- [ ] backup ฐานข้อมูลล่าสุดพร้อมใช้งาน
- [ ] ถ้ามีงานใหญ่ ได้วางแผน restore ไว้แล้ว

### 2.6 ตรวจไฟล์แนบและ storage

- [ ] อัปโหลดรูปโปรไฟล์ได้
- [ ] อัปโหลดลายเซ็นได้
- [ ] เปิดรูป/ลายเซ็นย้อนหลังได้
- [ ] เอกสารปิดงานยังดึงลายเซ็นจากโปรไฟล์ได้ถูกต้อง

### 2.7 ตรวจรายงานและเอกสาร

- [ ] Export Excel ใช้งานได้
- [ ] เอกสารปิดงานพิมพ์ได้
- [ ] เลขที่แจ้งซ่อมแสดงถูกต้อง
- [ ] ชื่อผู้รับงาน / ผู้ตรวจรับ / ลายเซ็นแสดงถูกต้อง

### 2.8 ตรวจ responsive

- [ ] หน้าแรกบนมือถือใช้งานได้
- [ ] ปุ่ม login / logout ไม่ล้น
- [ ] หน้า Dashboard บนมือถือยังอ่านได้
- [ ] หน้า Tracking บนมือถือยังดู timeline ได้
- [ ] หน้า Work List บนมือถือยังใช้งานได้

### 2.9 ตรวจ production config

- [ ] environment variables ครบ
- [ ] `DATABASE_URL` ถูกต้อง
- [ ] `DIRECT_URL` ถูกต้อง
- [ ] `SESSION_SECRET` ถูกต้อง
- [ ] ค่า Supabase Storage ถูกต้อง
- [ ] ไม่มีการชี้ไป dev database โดยผิดพลาด

### 2.10 ตรวจความพร้อมก่อนกดขึ้นจริง

- [ ] คุณอนุมัติให้ขึ้น production แล้ว
- [ ] มีคนรับรู้ว่ากำลังจะ deploy
- [ ] รู้ว่า deploy ครั้งนี้ต้องตรวจอะไรหลังขึ้นจริง

## 3. Checklist หลัง Deploy

หลัง deploy เสร็จ ให้ตรวจทันที:

- [ ] หน้าเว็บ production เปิดได้
- [ ] login ได้
- [ ] Dashboard โหลดได้
- [ ] All Work โหลดได้
- [ ] Request ใช้งานได้
- [ ] Tracking ใช้งานได้
- [ ] เปิดงานตัวอย่างในระบบได้
- [ ] พิมพ์เอกสารปิดงานได้
- [ ] ไม่มี error สำคัญบนหน้าเว็บ

## 4. ถ้า Release มีการเปลี่ยนข้อมูลจริง

ให้เช็กเพิ่ม:

- [ ] มี backup ก่อนเปลี่ยน
- [ ] รู้ว่าถ้าผิดจะ rollback อย่างไร
- [ ] มีไฟล์หรือ log อ้างอิงว่ารอบนี้แก้อะไรไปบ้าง

## 5. ถ้า Release เป็นงานด่วน (Hotfix)

ให้ใช้ checklist สั้นนี้:

- [ ] ยืนยันว่าปัญหาเกิดบน production จริง
- [ ] แก้เฉพาะจุดที่จำเป็น
- [ ] ทดสอบจุดที่แก้แล้ว
- [ ] ทดสอบไม่ให้กระทบ flow หลัก
- [ ] backup พร้อม
- [ ] deploy แล้วตรวจ production ทันที

## 6. สรุปขั้นตอน release ที่แนะนำ

ลำดับที่ควรใช้ทุกครั้ง:

1. ทำงานบน branch แยก
2. ทดสอบ local
3. push branch ขึ้น GitHub
4. ตรวจ preview
5. เช็กตาม checklist นี้
6. เมื่อพร้อมค่อย merge เข้า `master`
7. deploy production
8. ตรวจหลัง deploy ทันที

## 7. สรุปแบบสั้นที่สุด

ก่อนขึ้น production ต้องตอบให้ได้ว่า:

- งานนี้เสร็จจริงหรือยัง
- ทดสอบครบหรือยัง
- ข้อมูล production ปลอดภัยหรือยัง
- ถ้ามีปัญหา rollback ได้ไหม
- คุณอนุมัติแล้วหรือยัง

---

ถ้าทุกข้อสำคัญผ่านแล้ว ค่อยปล่อยขึ้น production จะปลอดภัยที่สุดสำหรับโปรเจกต์นี้
